# Правила форматирования итогового адаптированного резюме

Документ описывает фактические правила, по которым проект формирует итоговое адаптированное резюме в Google Docs.

## 1) Общая логика формирования

1. LLM возвращает текст резюме в Markdown.
2. Система копирует Google Doc шаблон.
3. Далее используется один из режимов вставки:
   - Секционный режим (предпочтительный): замена секционных плейсхолдеров.
   - Fallback-режим: замена через `{{CV_CONTENT}}`.
4. После вставки текста выполняется замена плейсхолдеров реквизитов кандидата (если есть данные).

## 2) Поддерживаемые плейсхолдеры шаблона

### 2.1 Секционные плейсхолдеры

Используются для точечной вставки разделов резюме:

- `{{SUMMARY}}`
- `{{SKILLS}}`
- `{{EXPERIENCE}}`
- `{{EDUCATION}}`
- `{{LANGUAGES}}`

Чтобы секционный режим сработал корректно, LLM-ответ должен содержать разделы:

- `## Summary`
- `## Skills`
- `## Experience`
- `## Education`
- `## Languages`

Важно:

- Если в документе найден хотя бы один секционный плейсхолдер, система выполняет замену по секциям.
- Если какая-то секция в markdown отсутствует, соответствующий плейсхолдер будет заменен на пустой текст.
- Для каждой секции применяется markdown-парсинг: заголовки, списки и inline-форматирование обрабатываются по тем же правилам, что и в режиме `{{CV_CONTENT}}`.
- Стиль абзаца всегда задается явно при вставке секции, чтобы не наследовать случайный стиль плейсхолдера (например, bullet).

### 2.2 Плейсхолдер полного контента (fallback)

- `{{CV_CONTENT}}`

Используется, если секционные плейсхолдеры в шаблоне не найдены.

Поведение:

- Если `{{CV_CONTENT}}` найден: удаляется только этот токен, и на его место вставляется весь markdown-контент (в формате plain text + стили).
- Если `{{CV_CONTENT}}` не найден: очищается тело документа и вставляется весь контент с начала документа.

Критично:

- `{{CV_CONTENT}}` нужно вводить в шаблоне одним непрерывным текстовым фрагментом.
- Если Google Docs разбил плейсхолдер на несколько text run, точный поиск может не сработать, и включится fallback с полной заменой body.

### 2.3 Плейсхолдеры реквизитов кандидата

Если в Master CV переданы данные applicant, в тексте допускаются плейсхолдеры:

- `{{NAME}}`
- `{{EMAIL}}`
- `{{PHONE}}`
- `{{LINKEDIN}}`

После генерации текста они заменяются реальными значениями через `replaceAllText` по всему документу (body/header/footer и др.).

### 2.4 Плейсхолдер названия позиции из вакансии

Поддерживается плейсхолдер:

- `{{Role as in JD}}`

Поведение:

- Значение берется из поля `job_title` текущей вакансии (как название позиции в JD/Tracker).
- Перед заменой выполняется очистка до базовой должности:
   - удаляются служебные хвосты вроде `with verification`;
   - удаляются скобочные уточнения вроде `(Contract)`, `(80%)`;
   - при наличии распознаваемой роли возвращается канонический вариант (`Principal Product Owner`, `Senior Product Owner`, `Product Owner`, `Product Manager`, `PMO`).
- Замена выполняется технически через `replaceAllText` на этапе пост-обработки Google Doc.
- Изменять системный промпт LLM для этого плейсхолдера не требуется.

## 3) Спецсимволы и Markdown-разметка, которые поддерживаются

### 3.1 Заголовки

Маппинг markdown-заголовков в named styles Google Docs:

- `# ...` -> `TITLE`
- `## ...` -> `HEADING_1`
- `### ...` -> `HEADING_2`
- `#### ...` и глубже -> `HEADING_3`

### 3.2 Списки

Буллеты создаются для строк:

- `- пункт`
- `* пункт`
- `+ пункт`
- `1. пункт`, `2. пункт`, ...

Все они конвертируются в параграфы списка Google Docs (`BULLET_DISC_CIRCLE_SQUARE`).

### 3.3 Inline-форматирование

Поддерживается:

- `***text***` -> bold + italic
- `**text**` -> bold
- `__text__` -> bold
- `*text*` -> italic
- `_text_` -> italic
- `` `text` `` -> маркеры удаляются, текст остается без monospace-стиля

### 3.4 Горизонтальные разделители

Строки вида ниже пропускаются и не вставляются как отдельный контент:

- `---`
- `***`
- `===` (три и более символов)

## 4) Обязательные ограничения для LLM-результата

Система ожидает, что LLM вернет:

1. Только текст резюме без пояснений.
2. Структуру с разделами в порядке:
   - `## Summary`
   - `## Skills`
   - `## Experience`
   - `## Education`
   - `## Languages`
3. Без дополнительных секций типа Hook/Cover Letter.

## 5) Практические рекомендации по шаблону Google Doc

1. Для максимального контроля верстки используйте секционные плейсхолдеры в нужных местах шаблона.
2. Если нужен единый блок вставки, оставьте `{{CV_CONTENT}}` в точке вставки.
3. Не редактируйте плейсхолдеры посимвольно разными стилями, чтобы не допустить разбиения на text run.
4. Если в шаблоне есть реквизиты кандидата (в шапке/подвале), используйте `{{NAME}}`, `{{EMAIL}}`, `{{PHONE}}`, `{{LINKEDIN}}`.

## 6) Пример минимального шаблона (секционный режим)

```text
{{NAME}}
{{EMAIL}} | {{PHONE}} | {{LINKEDIN}}

Summary
{{SUMMARY}}

Skills
{{SKILLS}}

Experience
{{EXPERIENCE}}

Education
{{EDUCATION}}

Languages
{{LANGUAGES}}
```

## 7) Чек-лист перед запуском

- В шаблоне есть либо секционные плейсхолдеры, либо `{{CV_CONTENT}}`.
- Плейсхолдеры написаны точно, в верхнем регистре, с двойными фигурными скобками.
- Системный промпт не требует лишних секций вне целевой структуры.
- При использовании реквизитов заполнены поля applicant в Master CV.
